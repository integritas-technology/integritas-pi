## Notes / Open Questions

- The plan (`docs/plans/minima-rpc-console.md`) named 7 commands to permanently exclude from the catalog (`vault`, `sendfrom`, `signfrom`, `createfrom`, `postfrom`, `quit`, `keys`) as a representative, not exhaustive, list, and explicitly asked for reconciliation against real `help` output during implementation. Reconciling turned up two more commands with the same risk profile that got added to the exclusion list: `createtokenfrom` (same "`*from`" family, requires a raw `privatekey:` argument like the four excluded siblings) and `decryptbackup` (can turn an encrypted backup file into plaintext seed/key material). Flagging this judgment call explicitly — worth the user's sanity check since it extends the plan's stated scope by two commands.
- `sphincs` and `maxsign` both take an *optional* `privatekey:`/similar param but were kept in the catalog as write/default-off (not excluded) — reasoning: unlike the `*from` family, they're general-purpose crypto utilities that don't need the wallet's own stored key to operate, so the double-gate (admin must explicitly enable + re-auth to edit the whitelist) was judged sufficient. Revisit if that reasoning doesn't hold up.
- Architecture principle carried over from earlier in this branch: backend is the sole source of truth for Minima operation state; this feature follows the same rule by dispatching `megammrsync`/`peers action:addpeers` through the existing tracked actions rather than a parallel RPC path.
